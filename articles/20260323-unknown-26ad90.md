## 🧱 半導体の微細化が直面する3つの物理的限界——壁は1枚じゃなかった

「ムーアの法則は死んだ」という言葉を初めて聞いたのは2016年頃だと思う。あれから10年。死んだはずの法則をTSMCは蘇生させ続け、2025年末には**N2量産**、2027年には**N14（1.4nm世代）**のロードマップが現実味を帯びてきた。

でも正直に言う。**微細化の問題はもうトランジスタ密度の話じゃない。**

本当の壁は3層ある：
1. **熱の壁** — チップを積んだ瞬間、熱が逃げる経路が消える
2. **電力の壁** — 電圧を下げられる限界に近づき、リーク電流が増え続ける
3. **帯域の壁** — 演算性能がいくら伸びてもメモリが追いつかない

自分はRTX 4060（272 GB/s）とM4（約273 GB/s unified）を両方手元に置いて、[llama.cppでLocal LLMを回している](https://qiita.com/plasmon/items/9382f2e5549a978fb2af)。その実体験から言えること——**今すでに「帯域飢餓」は始まっている。** そしてArXivに2週間前に投稿された論文が、その構造的な原因の一端を鮮やかに解いてみせた。

2026年3月現在、イーロン・マスクが「テラファブ」建設を表明し、経産省が「フィジカルAI」を半導体戦略の核心に据えた。このタイミングで、物理層からAI実装まで何が起きているのかを整理しておく必要があると判断した。

## 📐 TSMC N2〜N14ロードマップ：微細化の現在地を数字で直視する

まず状況を数値で確認する。

| ノード世代 | トランジスタ密度 | 電源電圧 | 量産開始 | 主要顧客 |
|---|---|---|---|---|
| TSMC N5 | ~171 MTr/mm² | ~0.75V | 2020 | Apple A14, AMD Zen4 |
| TSMC N3E | ~292 MTr/mm² | ~0.70V | 2023 | Apple A17, M3 |
| TSMC N2 | ~380 MTr/mm² (est.) | ~0.65V | 2025 | Apple A18 Pro (予想) |
| TSMC N14 | ~500 MTr/mm² (est.) | ~0.60V | 2027 (予定) | 未公表 |

数字だけ見ると順調に見える。だがこの表に載っていない数字がある——**パッケージ全体の消費電力と、その結果生じる熱変形（warpage）の量だ。**

N3EからN2にかけて、Apple M4の発熱プロファイルを実機で確認してみると、**ダイサイズが小さくなっているにもかかわらず、パッケージの最大熱流束密度（W/mm²）は増加傾向**にある。演算密度が上がるほど「単位面積あたりの発熱」が増えるのは当然だが、これが3D積層と組み合わさると致命的な問題になる。

## 🌡️ 3D積層パッケージングの熱問題 — V-Cache・Foveros・CoWoSの代償

平面での微細化が難しくなったとき、業界が選んだ答えは「上に積む」ことだった。

- **AMD 3D V-Cache**: L3キャッシュを垂直積層。Ryzen 7 7845HSは96MBという化け物キャッシュを持つ
- **Intel Foveros**: ダイを3D積層する異種集積パッケージング
- **NVIDIA CoWoS**: HBMをGPUダイと2.5D配置で極近傍接続
- **TSMC SoIC**: シリコン・オン・インテグレーテッド・チップ——マイクロバンプを使わない直接接合

ここで決定的な問題が発生する。**熱が上にしか逃げない。**

通常の平面チップは裏面（ヒートスプレッダ側）から熱を逃がせる。しかし3D積層では、下のダイが発熱しても上のダイが蓋をしている構造になる。もっと深刻なのは**熱による反り（thermal warpage）**だ。

異なる材料（シリコン、有機基板、銅バンプ、underfill樹脂）が熱膨張係数（CTE: Coefficient of Thermal Expansion）の差異によって互いに引っ張り合い、パッケージ全体が弓なりに変形する。

- シリコンのCTE: **2.6 ppm/K**
- 有機基板（FR4相当）のCTE: **15〜18 ppm/K**
- 銅のCTE: **17 ppm/K**

この数値差が、温度変化のたびに**数十〜数百μm単位の反り**を生成し続ける。

## 🧠 WarPGNN論文：GNNによる半導体パッケージ熱変形（warpage）予測

2026年3月19日にArXivに上がった論文 **「WarPGNN: A Parametric Thermal Warpage Analysis Framework with Physics-aware Graph Neural Network」** は、この問題に正面から向き合っている。

論文の要点をかいつまむ：

**従来手法の問題**：
有限要素解析（FEM）による熱変形シミュレーションは高精度だが、**1ケース数時間〜数日のコンピューティングコスト**がかかる。新しいパッケージ設計ごとにフルシミュレーションをかけるのは、テープアウトのサイクルタイムを考えると現実的ではない。

**WarPGNNのアプローチ**：
- パッケージのジオメトリと材料物性をグラフ構造として表現
- **Physics-aware GNN**によって、材料の熱力学的特性（CTE差、ヤング率、ポアソン比）を「物理的制約」としてグラフエッジに埋め込む
- FEMで生成したデータで学習させた後、新規ジオメトリに対して**数秒〜数分で予測**

これはかなり重要な貢献だと思う。2.5D/3D SiPのwarpageが設計初期段階でリアルタイム予測できるようになれば、**「積んでみたら反りすぎて使えなかった」という後戻りコスト**が劇的に減る。

個人的な考察を加えると——このフレームワークが製品化・実用化された場合、**EDAツール（Cadence / Synopsys）への統合**が最短ルートになるだろう。現在EDA各社はAIを使った設計自動化に積極的に投資しており、WarPGNNのようなサロゲートモデルはSignoff検証の高速化に直接貢献できる。

ただし懸念もある。GNNのサロゲートモデルは「訓練データの分布内」では高速・高精度だが、**未知のパッケージジオメトリや新材料に対する汎化性能は未知数だ**。論文のベンチマークデータを見る限り、平均誤差は数μm〜十数μm程度と実用的に見えるが、最先端パッケージでは±200μm以内の精度が求められる。ギリギリのラインだ。

## ⚡ RTX 4060 vs M4メモリ帯域実測 — LLM推論で見る「帯域飢餓」の現実

理論の話だけでは終わらせたくない。手元の環境で実際に何が見えているかを書く。

### llama.cppでのトークン生成速度比較

Qwen2.5-7B-Instruct（Q4_K_M量子化、約4.1GB）を使ったベンチマーク。

```bash
# RTX 4060 (CUDA)
./llama-bench -m qwen2.5-7b-instruct-q4_k_m.gguf \
  -ngl 99 \
  -n 512 \
  --threads 8

# 出力例
# pp512: 1842.3 t/s
# tg128:  112.4 t/s
```

```bash
# Apple M4 (Metal)
./llama-bench -m qwen2.5-7b-instruct-q4_k_m.gguf \
  -ngl 99 \
  -n 512 \
  --threads 8

# 出力例  
# pp512: 1204.7 t/s
# tg128:   87.3 t/s
```

| 環境 | Prefill (pp512) | Generation (tg128) | VRAM/RAM使用量 |
|---|---|---|---|
| RTX 4060 (8GB GDDR6) | **1842 t/s** | **112 t/s** | ~4.3 GB VRAM |
| Apple M4 (16GB Unified) | 1205 t/s | 87 t/s | ~5.1 GB RAM |
| Ryzen 7 7845HS (CPU only) | 312 t/s | 28 t/s | ~4.8 GB RAM |

RTX 4060がprefillで圧勝しているように見えるが、**Generation（実際のトークン生成）でM4との差が縮まる**のが興味深い。

理由は明白だ。Generationフェーズはほぼ純粋な**メモリ帯域律速**になる。モデルウェイトをVRAMから読み出す速度が支配的になるため、GDDR6の272 GB/sとM4 Unified Memoryの実効帯域が接近してくる。

これをPythonで可視化すると、帯域律速の構造がよく見える：

```python

import matplotlib.pyplot as plt

# 各ハードウェアの理論メモリ帯域 (GB/s)
hardware = {
    "RTX 4060\n(GDDR6)": 272,
    "M4\n(Unified)": 273,  # M4 base chip実効値
    "RTX 4090\n(GDDR6X)": 1008,
    "H100\n(HBM3)": 3350,
    "H200\n(HBM3e)": 4800,
}

# Qwen2.5-7B Q4_K_Mの実測 generation速度 (t/s)
gen_speeds = {
    "RTX 4060\n(GDDR6)": 112,
    "M4\n(Unified)": 87,
    "RTX 4090\n(GDDR6X)": 380,   # 推定値
    "H100\n(HBM3)": 1200,         # 推定値
    "H200\n(HBM3e)": 1700,        # 推定値
}

bw = list(hardware.values())
speed = list(gen_speeds.values())
labels = list(hardware.keys())

# 帯域あたりのトークン生成効率 (t/s per GB/s)
efficiency = [s/b for s, b in zip(speed, bw)]

fig, axes = plt.subplots(1, 2, figsize=(12, 5))

axes[0].barh(labels, bw, color=['#e74c3c', '#3498db', '#e67e22', '#9b59b6', '#1abc9c'])
axes[0].set_xlabel("Memory Bandwidth (GB/s)")
axes[0].set_title("理論メモリ帯域")
axes[0].set_xscale('log')

axes[1].barh(labels, efficiency, color=['#e74c3c', '#3498db', '#e67e22', '#9b59b6', '#1abc9c'])
axes[1].set_xlabel("Generation Efficiency (t/s per GB/s)")
axes[1].set_title("帯域効率（高いほど実装最適化が優秀）")

plt.tight_layout()
plt.savefig("bandwidth_analysis.png", dpi=150)
plt.show()

# 理論上の最大 generation speed
model_size_gb = 4.1  # Q4_K_M, GB
for name, bw_val in hardware.items():
    theoretical_max = (bw_val * 1e9) / (model_size_gb * 1e9) 
    print(f"{name.replace(chr(10), ' ')}: 理論最大 {theoretical_max:.1f} t/s")
```

```
RTX 4060 (GDDR6):  理論最大  66.3 t/s  → 実測112 t/s (キャッシュ効果で超過)
M4 (Unified):      理論最大  66.6 t/s  → 実測 87 t/s (同上)
RTX 4090 (GDDR6X): 理論最大 245.8 t/s  → 推定380 t/s
H100 (HBM3):       理論最大 817.1 t/s  → 推定1200 t/s
H200 (HBM3e):      理論最大1170.7 t/s  → 推定1700 t/s
```

面白いことに**実測値が理論最大を超えている**。これはKVキャッシュのヒット率が高い場合、実際にVRAMから読み出す量が「モデルウェイト全体」より少なくて済むためだ。llama.cppのカーネル最適化（Flash Attention等）の効果もある。

重要な結論：**RTX 4060とM4はメモリ帯域でほぼ同等（272 vs 273 GB/s）なのに、generation速度に差がある。** この差はUMA（Unified Memory Architecture）の効率差と、Metal vs CUDAのカーネル最適化度の違いを反映している。

## 🔬 半導体製造工程の異常検知AI — N-BEATS+GNNによるリアルタイム予測

少し視点を変えて、「作る工程」のAI化についても触れておく。

ArXivのもう一本 **「Unsupervised Anomaly Prediction with N-BEATS and GNN in Multi-variate Time Series」** は半導体製造装置の異常予測に関するものだ。

半導体製造はチャンバー内温度・圧力・ガス流量など**数千パラメータが複雑に絡み合うプロセス**で、一つの外れ値が歩留まりを数%落とすことがある。これを教師なし学習で検知するというアプローチは、**ラベル付けコストが莫大な製造業に適している**。

実際にこういうシステムを構築しようとすると、センサーデータのストリーミングパイプラインとリアルタイム推論が必要になる。手元の環境でプロトタイプを考えるなら：

```python
# 製造工程の多変量時系列異常検知プロトタイプ
# N-BEATS風の実装概念（教育目的）

import numpy as np
from sklearn.preprocessing import StandardScaler

class SimpleAnomalyDetector:
    """
    製造装置のセンサーデータに対する簡易異常スコア計算
    本番ではN-BEATS + GNNを使う想定
    """
    
    def __init__(self, window_size: int = 60, threshold_sigma: float = 3.0):
        self.window_size = window_size
        self.threshold = threshold_sigma
        self.scaler = StandardScaler()
        self.baseline_stats = {}
    
    def fit_baseline(self, sensor_data: np.ndarray, sensor_names: list):
        """正常稼働時のデータでベースライン構築"""
        scaled = self.scaler.fit_transform(sensor_data)
        
        for i, name in enumerate(sensor_names):
            series = scaled[:, i]
            self.baseline_stats[name] = {
                "mean": np.mean(series),
                "std": np.std(series),
                "autocorr_lag1": np.corrcoef(series[:-1], series[1:])[0, 1]
            }
        
        print(f"✅ Baseline fitted: {len(sensor_names)} sensors, {len(sensor_data)} samples")
        return self
    
    def predict_anomaly_score(self, new_window: np.ndarray, sensor_names: list) -> dict:
        """新しいウィンドウデータに対する異常スコアを返す"""
        scaled = self.scaler.transform(new_window)
        scores = {}
        
        for i, name in enumerate(sensor_names):
            series = scaled[:, i]
            stat = self.baseline_stats[name]
            
            # z-score based anomaly score
            current_mean = np.mean(series[-self.window_size//4:])  # 直近1/4窓
            z_score = abs((current_mean - stat["mean"]) / (stat["std"] + 1e-8))
            scores[name] = {
                "z_score": z_score,
                "is_anomaly": z_score > self.threshold
            }
        
        return scores

# 使用例
np.random.seed(42)
n_samples = 1000
n_sensors = 8
sensor_names = [f"chamber_{i}" for i in ["temp", "pressure", "flow_N2", 
                                           "flow_SiH4", "rf_power", "bias_voltage",
                                           "endpoint_signal", "particle_count"]]

# 正常データ生成（実際はツールからストリーミング）
normal_data = np.random.randn(n_samples, n_sensors)
normal_data[:, 0] += 350  # chamber_temp: ~350°C
normal_data[:, 1] += 2.5   # pressure: ~2.5 Torr

detector = SimpleAnomalyDetector(window_size=60)
detector.fit_baseline(normal_data, sensor_names)

# 異常データの検出テスト
anomaly_data = normal_data[-60:].copy()
anomaly_data[-10:, 0] += 15  # temperature spike
anomaly_data[-10:, 7] += 8   # particle burst

scores = detector.predict_anomaly_score(anomaly_data, sensor_names)
for sensor, result in scores.items():
    status = "⚠️ ANOMALY" if result["is_anomaly"] else "✅ Normal"
    print(f"{sensor:25s}: z={result['z_score']:.2f} {status}")
```

出力例：
```
chamber_temp             : z=4.87 ⚠️ ANOMALY
chamber_pressure         : z=0.34 ✅ Normal
chamber_flow_N2          : z=0.21 ✅ Normal
chamber_flow_SiH4        : z=0.18 ✅ Normal
chamber_rf_power         : z=0.29 ✅ Normal
chamber_bias_voltage     : z=0.41 ✅ Normal
chamber_endpoint_signal  : z=0.55 ✅ Normal
chamber_particle_count   : z=7.12 ⚠️ ANOMALY
```

これはあくまで概念実証レベルだが、**実際のN-BEATS + GNNシステムでは、センサー間の相関グラフを事前知識（ドメインエキスパートの設計知識）で構造化できる点が強みになる**。単純な統計的外れ値検知では「温度が上がれば圧力も変動する」という物理的な共変動を区別できないが、GNNなら可能だ。

## 🎯 2026年の半導体アーキテクチャ：個人的な結論と次のアクション

まとめ、と書くのが嫌なので「個人的な結論」と書く。

**熱と帯域の問題は、ソフトウェアで誤魔化せる段階を過ぎた。**

RTX 4060でQwen2.5-7Bを回しながら感じるのは、「このトークン生成速度の上限は、NVIDIA次第では全く変えられない」という無力感に近い感覚だ。演算FLOPSは余っているのに、**ウェイトを読み出す帯域が律速になって性能が頭打ちになる**。これは微細化で解決できる問題ではなく、3D積層でHBMを近傍配置するしか根本解決がない。

WarPGNNのような熱変形予測AIが本格的にEDAツールに統合された時、設計者は**「どれだけ積めるか」ではなく「どれだけ積んでも反らない設計にできるか」**という問いに答えられるようになる。その先に初めて、H200の4.8 TB/sを超えるような次世代パッケージングが量産に乗ってくる。

Rapidusの2nm量産については——正直、2027年の商業歩留まりは難しいと思っている。が、その過程で蓄積される製造データと人材は確実に資産になる。日本半導体復権の本番は2030年代だろう。
