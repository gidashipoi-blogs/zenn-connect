## Speculative Decodingで27Bが逆に遅くなった

> 本記事の数値はすべて筆者環境（Ryzen 7 7845HS / 32GB DDR5 / RTX 4060 Laptop 8GB）での実測値です。

---

## Speculative Decodingという甘い誘惑

「小さいモデルで下書きして、大きいモデルで検証する。大きいモデルの品質を、小さいモデルに近い速度で得られる。」

Speculative Decoding（投機的デコーディング）のセールストークはこうだ。2022年のDeepMind論文（Leviathan et al.）以降、llama.cppにも実装され、`-md`（model-draft）フラグひとつで使える。理屈上は夢のような技術だ。

自分の環境を見てみる:

- **Qwen3.5-27B** (dense): 3.5 t/s（ngl=24、普段使い）
- **Qwen3.5-9B**: 33 t/s（ngl=99、全レイヤGPU）

27Bの品質を、33 t/sに近い速度で得られるなら最高だ。9Bで8トークン先に下書きして、27Bがまとめて検証する。受理率が高ければ、1回の検証で複数トークンが確定する。理論上は**2〜3倍の高速化**が狙える。

ドラフトモデルのサイズを0.8B / 4B / 9Bの3パターンで試した。結果は全部逆だった。

---

| コンポーネント | スペック |
|---|---|
| CPU | Ryzen 7 7845HS (8C/16T) |
| RAM | 32GB DDR5-4800 |
| GPU | RTX 4060 Laptop 8GB GDDR6 (272 GB/s) |

| 主モデル | Qwen3.5-27B-Q4_K_M (~16.5GB) |
| ドラフトモデル | Qwen3.5-0.8B / 4B / 9B-Q4_K_M |

## VRAM 8GBの現実：両方積めない

ここが最初の壁だ。Speculative Decodingは**主モデルとドラフトモデルを同時にメモリに載せる必要がある**。

```
必要VRAM概算（27B + 9Bの場合）:
  9B (Q4_K_M)  ≈ 5.5 GB
  27B (Q4_K_M) ≈ 16.5 GB
  KVキャッシュ×2 ≈ 0.5〜1 GB
  ─────────────────────────
  合計         ≈ 22.5 GB   ← RTX 4060の8GBの約3倍
```

物理的に不可能。普段の27B単体運用でも、ngl=24（48層中24層をGPUに載せる）が限界だった。2モデル同時なんて論外——と最初は思った。

### 解法：VRAM配分を最適化する

llama.cppの`-ngl`（主モデルのGPUレイヤ数）と`-ngld`（ドラフトのGPUレイヤ数）は別々に指定できる。つまり、8GBのVRAMを「主モデルの一部 + ドラフト全体」に分割できる。

ドラフトモデルが小さいほど、主モデルにVRAMを多く回せる:

```
ドラフト0.8B (0.6GB) → 27Bに7.4GB → ngl=20（48層中20層GPU）
ドラフト4B   (2.5GB) → 27Bに5.5GB → ngl=13（48層中13層GPU）
ドラフト9B   (5.5GB) → 27Bに2.5GB → ngl=0 （完全CPU）
```

```bash
# 例: 0.8Bドラフト構成
llama-cli \
  -m  Qwen3.5-27B-Q4_K_M.gguf \    # 主モデル
  -md Qwen3.5-0.8B-Q4_K_M.gguf \   # ドラフトモデル
  -ngl 20 \                         # 27B: 20レイヤGPU
  -ngld 99 \                        # 0.8B: 全レイヤGPU
  --draft-max 8 \                   # 1回の投機で最大8トークン
  -c 2048 -t 8 -n 256 --temp 0.0
```

ここにトレードオフがある。**ドラフトが大きい＝受理率は高いがGPUレイヤが減る。ドラフトが小さい＝GPUレイヤは増えるが受理率が下がる。** どこが最適解なのか？ 3パターン全部試した。

---

## 実測結果：3パターン全敗

```
テスト1: 27B (ngl=20) + 0.8B draft (ngld=99)
  VRAM: 0.8B ~0.6GB + 27B 20層 ~6.5GB ≈ 7.1GB
  Generation:   3.1 t/s    ← ベースラインより11%遅い

テスト2: 27B (ngl=13) + 4B draft (ngld=99)
  VRAM: 4B ~2.5GB + 27B 13層 ~4.5GB ≈ 7.0GB
  Generation:   2.8 t/s    ← 20%遅い

テスト3: 27B (ngl=0) + 9B draft (ngld=99)
  VRAM: 9B ~5.5GB + 27B 0層 = 5.5GB
  Generation:   2.5 t/s    ← 29%遅い

ベースライン: 27B通常 (ngl=24)
  Generation:   3.5 t/s

参考: 9B単体 (ngl=99)
  Generation:  33 t/s
```

| 構成 | ドラフト | 27B GPU層 | 速度 | vs ベースライン |
|---|---|---|---|---|
| 27B + 0.8B | 0.6GB | 20/48 | **3.1 t/s** | -11% |
| 27B + 4B | 2.5GB | 13/48 | **2.8 t/s** | -20% |
| 27B + 9B | 5.5GB | 0/48 | **2.5 t/s** | -29% |
| 27B 通常 | — | 24/48 | **3.5 t/s** | baseline |
| 9B 単体 | — | 全層 | **33 t/s** | — |

**3パターン全てベースラインに負けた。** ドラフトが小さい方がマシだが、それでも通常の27Bより遅い。

---

## なぜ全パターン遅くなるのか：VRAMの奪い合い

3パターンの結果が綺麗に並んでいるのが重要だ。ドラフトが大きくなるほど遅くなっている。

### パターン分析：二重の罠

**ドラフトを大きくする方向（9B）:**
9Bは受理率が高いはずだが、5.5GBのVRAMを食う。27Bは完全にCPU落ち（ngl=0）。nvidia-smiを見ると:

```
GPU Utilization:  5〜10%
VRAM Used:        ~7140 MiB
GPU Temperature:  42°C（ほぼアイドル）
```

GPUがほとんど仕事をしていない。27Bの検証がCPUで数秒かかり、その間GPUはただ待つ。

**ドラフトを小さくする方向（0.8B）:**
0.8Bは0.6GBしか食わないから、27Bに20層分のGPUを残せる。通常運用（ngl=24）に近い。ただし0.8Bと27Bでは能力差が大きすぎて、受理率が下がる。8トークン投機しても2〜3トークンしか受理されないなら、検証のオーバーヘッド分だけ損する。

**中間の4B:**
両方の弱点を中途半端に受ける。

```
時間軸 → （9Bドラフトの場合）
GPU:  [draft 8tok] [idle........] [draft 8tok] [idle........]
CPU:  [idle]       [verify 8tok ] [idle]       [verify 8tok ]
                   ↑ ここが律速
```

つまり8GB VRAMでは**どっちに振ってもダメ**だ。大きいドラフトはGPU層を奪い、小さいドラフトは受理率が足りない。通常の27B（ngl=24）が「VRAM全部を自分で使える」構成に、分割した構成が勝てない。

### DRAMバンド幅が本当の敵

もう少し深掘りする。27BをCPU推論する場合、ボトルネックはCPU演算力ではなく**メモリバンド幅**だ。

```
DDR5-4800 (デュアルチャネル):
  理論バンド幅: 76.8 GB/s

Qwen3.5-27B-Q4_K_M:
  モデルサイズ: ~16.5 GB
  1トークン生成に必要な読み出し: ~16.5 GB
  → 理論最大速度: 76.8 / 16.5 ≈ 4.7 t/s

RTX 4060 GDDR6:
  理論バンド幅: 272 GB/s
  → 理論最大速度: 272 / 16.5 ≈ 16.5 t/s（3.5倍高速）
```

GPU（272 GB/s）からCPU（76.8 GB/s）にモデルを移動した時点で、帯域幅が**3.5分の1**に落ちる。speculative decodingの投機的並列処理で稼ぐはずの速度ゲインが、CPUメモリバンド幅の制約で完全に打ち消される。

実測の2.5 t/sが理論値4.7 t/sに届かないのは、KVキャッシュの読み書きやOSのメモリ管理オーバーヘッドが加わるためだ。むしろ理論値の53%出ているのは妥当な数字と言える。

## そもそもこのアルゴリズムは何を前提に設計されたか

物理的な「なぜ遅いか」は見えた。ここで一歩引いて、speculative decodingの**設計思想**を考える。

このアルゴリズムの原論文（Leviathan et al., 2022）が想定している環境は、**主モデルとドラフトモデルが同じ高速メモリ上に同居し、ドラフト生成と検証が交互に高速実行される**構成だ。GPU VRAMの中で完結するから、ドラフト→検証のラウンドトリップが高速に回る。

8GBでこの前提が崩れると、アルゴリズムの構造自体が破綻する。VRAMを分割した時点で主モデルの一部がCPUに落ち、検証ステップが低速メモリに律速される。投機的並列処理で稼ぐはずのゲインが、メモリ階層を跨いだ時点で消失する。**帯域幅が遅いから遅いのではなく、「同一メモリ空間での高速ラウンドトリップ」という設計前提が成立しないから遅い。**

この実験からわかるのは、**speculative decodingは設計前提の環境でしか機能しない**ということだ。

### 機能する条件

```
条件1: 主モデルとドラフトモデルがVRAMに同時収容できる
条件2: ドラフトモデルの速度 >> 主モデルの速度（最低5倍以上）
条件3: 受理率が十分高い（同一ファミリーの大小で70%以上が目安）
```

RTX 4060 8GBで条件1を満たそうとすると:

| 組み合わせ | 27B GPU層 | ドラフトVRAM | 実測 | 判定 |
|---|---|---|---|---|
| 27B + 0.8B | 20/48層 | 0.6 GB | 3.1 t/s | ❌ ベースライン以下 |
| 27B + 4B | 13/48層 | 2.5 GB | 2.8 t/s | ❌ 同上 |
| 27B + 9B | 0/48層 | 5.5 GB | 2.5 t/s | ❌ 最悪 |
| 7B + 1.5B (仮) | 全層 | ~1.0 GB | 未測定 | ⚠️ 7Bの品質でいいなら |

今回の実測で、8GBでは「どのドラフトサイズでも27Bのベースラインに勝てない」ことが確定した。7B + 1.5Bなら両方GPUに載るが、それなら9Bを単体で33 t/sで走らせた方が速くて賢い。

### 本来の適用領域

Speculative Decodingが真価を発揮するのは:

- **A100 80GB / H100 80GB**: 70B + 7B が余裕で同居
- **RTX 4090 24GB**: 27B + 7B や 14B + 1.5B が現実的
- **M4 Max 128GB Unified Memory**: 大容量UMAで大型モデルペアが可能

コンシューマGPU 8GBは、そもそもこの技術の想定ターゲットではない。

## 8GB VRAMの最適解はMoEだった

ここで前回の記事のデータを振り返る。

| モデル | 方式 | 速度 | VRAM | 品質感 |
|---|---|---|---|---|
| Qwen3.5-9B | dense, ngl=99 | **33 t/s** | 7.1 GB | ◎ |
| Qwen3.5-35B-A3B | MoE, partial | **8.6 t/s** | ~7 GB | ◎◎ |
| Qwen3.5-27B | dense, ngl=24 | 3.5 t/s | ~6.5 GB | ◎◎ |
| 27B + 0.8B speculative | GPU+GPU | **3.1 t/s** | ~7.1 GB | ◎◎ |
| 27B + 4B speculative | GPU+GPU | **2.8 t/s** | ~7.0 GB | ◎◎ |
| 27B + 9B speculative | CPU+GPU | **2.5 t/s** | ~7.1 GB | ◎◎ |

27Bの品質を速く得る手段として:

- **Speculative (最良の0.8Bドラフト構成でも)**: 3.1 t/s（通常27Bより遅い）
- **MoE (35B-A3B)**: 8.6 t/s（27Bと同等以上の品質、3倍速い）

8GB VRAM環境では、speculative decodingより**MoE（Mixture-of-Experts）モデルの方が圧倒的に効率がいい**。MoEは設計段階から「パラメータ総数は大きいが、推論時のアクティブパラメータは小さい」という構造を持っている。これはまさに「大きいモデルの品質を、小さいモデルの速度で」という、speculative decodingと同じ目標を、アーキテクチャレベルで解決している。

```
Speculative Decoding:
  推論時 = 2モデル同時ロード → VRAM圧迫
  ゲイン = 投機的並列化（CPU上では無意味）

MoE:
  推論時 = 1モデル、アクティブパラメータのみ計算
  ゲイン = アーキテクチャ自体が効率的
```

---

## Speculative Decodingが復活する日

この記事の結論は「speculative decodingは8GB VRAMでは使えない」だが、**技術自体が死んでいるわけではない**。

1. **VRAM容量の増加**: RTX 5060が12〜16GB VRAMになれば、27B + 3B程度の組み合わせが現実的に
2. **より小さいドラフトモデル**: 今回0.8Bまで試したが、それでも-11%。SmolLM2-135Mのような超軽量モデルならVRAM消費がほぼゼロになるが、受理率はさらに下がる。異なるアーキテクチャ間のspeculativeは現状サポートが限定的
3. **ハードウェア対応**: NPUやAMD XDNAのような専用推論エンジンがドラフト生成を担当する未来

特に2の「超小型ドラフトモデル」は、同一アーキテクチャファミリーの0.5Bモデルが充実してきた今、最も現実的な改善方向だ。ただし極小モデルは受理率が下がるため、そのトレードオフの検証が必要になる。

## 8GBの壁の前では、魔法は通用しない

```
RTX 4060 8GBでQwen3.5-27Bのspeculative decodingを3パターン実測:

  27B + 0.8B draft:  3.1 t/s（-11%）
  27B + 4B draft:    2.8 t/s（-20%）
  27B + 9B draft:    2.5 t/s（-29%）
  27B 通常:          3.5 t/s（baseline）

8GB VRAMを2モデルで分割した時点で、
通常運用の「VRAMを独占する27B」に勝てない。
MoE (35B-A3B: 8.6 t/s) が正解。
```

ドラフトモデルの大小を3パターン試して、トレードオフの両端を確認して、それでも全敗。これは「まだ最適な設定を見つけていない」のではなく、**8GB VRAMという物理制約の下ではspeculative decodingが構造的に機能しない**ということだ。

正直、2日かけて全パターン試した末の全敗はかなり悔しい。当面は素直にMoE 35B-A3B（8.6 t/s）を使う。ただ、一つ引っかかっているのはQwen3.5のthinkingモードだ。投機的に生成したトークンのうち、thinkingの`<think>...</think>`部分は受理率が通常テキストと違う可能性がある。thinkingが絡まない高性能モデル——Mistral系やLlama系——で同じ実験をやったら、結果が変わるかもしれない。次の実験候補として頭に入れておく。
