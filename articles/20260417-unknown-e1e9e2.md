# LLMを1つだけ使うのは8GBの無駄遣いだった

RTX 4060 8GBでローカルLLMを動かす。多くの人が「一番大きいモデルを1つ載せる」という運用をしている。Qwen2.5-32B Q4_K_Mは約20GBのモデルだが、ngl=60で部分オフロードすればVRAM 7.6GBで動く。残りはCPU/RAM。VRAMのほぼ全てを1つのモデルに使い切る運用だ。

この運用は効率的に見えて、実は非効率だ。

RouteLLM（arXiv:2406.18665, Ong et al., ICLR 2025）は**強いモデルと弱いモデルを組み合わせてルーティングすることで、品質を落とさずコストを2倍以上削減**できることを示した。FrugalGPT（arXiv:2305.05176, Chen et al.）はLLMカスケードで**GPT-4相当の精度を98%のコスト削減で達成**。Hybrid LLM（arXiv:2404.14618, Ding et al., ICLR 2024）は**大モデルの呼び出しを40%削減し、品質低下は1%未満**だった。

8GBのVRAMを1つのモデルに全振りするのではなく、用途別に複数の小モデルを使い分ける。この考え方を実装する。

---

## なぜ1モデル運用は非効率なのか

### タスクの大半は32Bを必要としない

以下はローカルLLMの典型的な利用パターンの概算（実際のタスク分布は人による）:

| タスク | 割合 (概算) | 必要な能力 | 適切なモデルサイズ |
|-------|------|-----------|------------------|
| function calling / ツール選択 | ~35% | JSON出力、関数名選択 | 4B (~2.5GB) |
| コード補完 / 短い生成 | ~25% | 文法理解、パターン補完 | 4-8B (~3-5GB) |
| 質問応答 / 知識回答 | 20% | 世界知識、推論 | 14-32B (7-18GB) |
| 長文要約 / 分析 | 15% | 長コンテキスト理解 | 14B+ (9GB+) |
| 創造的生成 / 複雑な推論 | 5% | 深い推論、創造性 | 32B+ (18GB+) |

タスクの60%は4-8Bモデルで十分。32Bが本当に必要なのは20%程度。残り80%のタスクに32Bを使うのは、ネジを回すのにハンマーを使うようなものだ。

### 32B 1モデル vs マルチモデル構成の比較

**シングルモデル構成**: Qwen2.5-32B Q4_K_M（モデル~20GB、ngl=60でVRAM 7.6GB使用、残りCPU/RAM）。全タスク統一で~10.8 t/s（ngl=60時実測）。function calling精度は80%前後（ベンチマーク記事の同クラス実績から推定）。余剰VRAMは0.4GBで他のモデルは載らない。

**マルチモデル構成**:

| 役割 | モデル | モデルサイズ | VRAM(全層GPU+KV) | 速度 | 備考 |
|------|--------|------------|------------------|------|------|
| ツール特化 | Qwen3.5-4B Q4_K_M | ~2.5GB | ~3-4GB (推定) | ~50 t/s (推定) | function calling、分類、ルーティング |
| 汎用 | Qwen3 8B Q4_K_M | ~4.7GB | ~6-7GB (推定) | ~25 t/s (推定) | 質問応答、コード補完、一般タスク |

同時起動はできない（合計9-11GB、8GBに収まらない）。4Bを常駐させ、8Bはオンデマンドでロードする排他運用。モデル切替のロード待ち（10-15秒）が最大のデメリットだ。

## 研究が裏付けるマルチモデルの優位性

「大モデル1つより、大小混合の方が効率的」という主張は、複数の査読済み論文で裏付けられている。

| 論文 | 手法 | 主要結果 |
|------|------|---------|
| RouteLLM (Ong et al., ICLR 2025) | 強/弱モデル間のルーティング | 品質を落とさずコスト2倍以上削減 |
| FrugalGPT (Chen et al., 2023) | LLMカスケード（安→高の段階的呼び出し） | GPT-4相当の精度を98%コスト削減で達成、または同コストで+4%精度 |
| Hybrid LLM (Ding et al., ICLR 2024) | 小/大モデルの二値ルーティング | 大モデル呼び出し40%削減、品質低下1%未満 |

これらはAPIベースのコスト最適化研究だが、核心のアイデアはローカル1GPU環境にも適用できる: **簡単なタスクに大モデルを使うのは無駄であり、適切にルーティングすれば品質を維持したまま処理速度を大幅に改善できる。**

ただし重要な違いがある。API環境ではモデル切替はAPI呼び先を変えるだけでコストゼロだが、ローカル1GPU環境ではモデルのアンロード→ロードに10-15秒かかる。この切替コストがローカル環境固有のトレードオフになる。

### なぜ混合が単一に勝つのか

直感に反する結果だが、理由は明確だ。

**1. 簡単なタスクに大モデルを使うと遅い。** 32B（部分オフロード、~10 t/s）で "Hello" に返事させるより、4B（全層GPU、推定~50 t/s）で即答させた方が5倍速い。

**2. 大モデルは簡単なタスクで過剰推論する。** パラメータが多いほど「考えすぎる」傾向がある。function callingでは小モデルが大モデルを上回るケースがある（ただしツール環境への依存性あり、別記事で詳述）。

**3. 小モデルは特化タスクで大モデルに匹敵する。** 構造化出力、分類、ルーティングは4Bで十分。小モデルの不得意（知識回答、長文推論）だけ大モデルに回す。

## 8GBで実装するマルチモデル構成

### 構成A: 常駐 + オンデマンド切替

```python
# llama-server を2段構成で運用
import subprocess
import requests
import json

class MultiModelRouter:
    def __init__(self):
        # 常駐: 軽量モデル (ルーティング + function calling)
        self.resident_model = "qwen3.5-4b-q4_k_m.gguf"  # 3.4GB
        self.resident_port = 8080
        self.resident_proc = None

        # オンデマンド: 重量モデル (知識回答、長文)
        self.heavy_model = "qwen2.5-14b-q4_k_m.gguf"    # ~8.7GB, 部分オフロード必須
        self.heavy_port = 8081
        self.heavy_proc = None

    def start_resident(self):
        """常駐モデルを起動 (VRAM ~3-4GB)"""
        self.resident_proc = subprocess.Popen([
            "llama-server",
            "-m", self.resident_model,
            "--port", str(self.resident_port),
            "-ngl", "99",  # 全層GPU
            "-c", "4096",
        ])

    def route_request(self, user_message: str) -> str:
        """リクエストを分類して適切なモデルに振り分ける"""
        # Step 1: 常駐モデルでタスク分類
        classification = self._classify(user_message)

        if classification in ["function_call", "simple_qa", "code_complete"]:
            # 常駐モデルで処理 (高速)
            return self._call_resident(user_message)
        else:
            # 重量モデルをロードして処理
            return self._call_heavy(user_message)

    def _classify(self, message: str) -> str:
        """タスク分類 (常駐モデルで実行)"""
        prompt = f"""Classify this request into one category:
- function_call: needs tool/API call
- simple_qa: factual question, short answer
- code_complete: code completion/generation
- knowledge: requires deep knowledge or reasoning
- long_context: requires reading/summarizing long text

Request: {message}
Category:"""

        resp = requests.post(
            f"http://localhost:{self.resident_port}/completion",
            json={"prompt": prompt, "n_predict": 20, "temperature": 0.1}
        )
        return resp.json()["content"].strip().lower()

    def _call_resident(self, message: str) -> str:
        """常駐モデルで応答"""
        resp = requests.post(
            f"http://localhost:{self.resident_port}/completion",
            json={"prompt": message, "n_predict": 500}
        )
        return resp.json()["content"]

    def _call_heavy(self, message: str) -> str:
        """重量モデルをロードして応答"""
        if not self.heavy_proc:
            self.heavy_proc = subprocess.Popen([
                "llama-server",
                "-m", self.heavy_model,
                "--port", str(self.heavy_port),
                "-ngl", "24",  # 部分GPU (VRAM制約)
                "-c", "8192",
            ])
            # ロード待ち (14Bで10-15秒)
            import time; time.sleep(15)

        resp = requests.post(
            f"http://localhost:{self.heavy_port}/completion",
            json={"prompt": message, "n_predict": 1000}
        )
        return resp.json()["content"]
```

### 構成B: 同時常駐 (4B + Embedding)

| コンポーネント | モデル | VRAM (推定) | 役割 |
|-------------|--------|------------|------|
| 推論 | Qwen3.5-4B Q4_K_M | ~3-4GB | function calling + 応答生成 |
| Embedding | BGE-M3 (sentence-transformers) | ~1.5-3.4GB※ | ベクトル検索 (ChromaDB) |
| **合計** | | **~5-7GB** | 残り1-3GB (KVキャッシュ + OS) |

※BGE-M3のVRAMはバッチサイズと系列長に依存。小バッチなら~1.5GB、大バッチでは~3.4GBまで増加する。

VRAM余裕は1-3GBとタイトだが、KVキャッシュのコンテキスト長を4096以下に制限すればAgentic RAGが1 GPUで完結する構成になる。

### 構成C: 3モデル切替 (タスク特化)

| 役割 | モデル | VRAM | 得意 | ロード時間 |
|------|--------|------|------|-----------|
| ツール特化 | Qwen3.5-4B Q4_K_M | ~3-4GB (全層GPU) | function calling, JSON生成, 分類 | ~3秒 |
| コード特化 | DeepSeek-Coder-V2 Lite 16B Q4_K_M | ~7GB (部分オフロード) | コード生成、デバッグ | ~10秒 |
| 知識特化 | Qwen2.5-14B Q4_K_M | ~7GB (部分オフロード) | 知識回答、長文要約、推論 | ~10秒 |

同時起動はしない。タスクに応じてアンロード→ロードする。ロード時間10秒のコスト vs 不適切なモデルで低品質な回答が返るコスト。10秒待つ方が安い。

## ルーティングの精度がすべてを決める

マルチモデル構成の弱点は、ルーティングの判断ミスだ。知識回答を4Bに回せば低品質な答えが返り、function callingを14Bに回せば遅くなる。

### 簡易ルーター実装

```python
import re

def simple_router(message: str) -> str:
    """ルールベースのタスクルーター"""

    # function calling パターン
    tool_patterns = [
        r"天気", r"検索", r"計算", r"変換",
        r"weather", r"search", r"calculate",
        r"ファイル.*開", r"API.*呼",
    ]
    if any(re.search(p, message) for p in tool_patterns):
        return "tool_specialist"

    # コードパターン
    code_patterns = [
        r"```", r"def ", r"class ", r"import ",
        r"コード", r"実装", r"バグ", r"エラー",
        r"code", r"implement", r"debug", r"fix",
    ]
    if any(re.search(p, message) for p in code_patterns):
        return "code_specialist"

    # 長文パターン
    if len(message) > 2000:
        return "knowledge_specialist"

    # 短い質問はツールモデルで十分
    if len(message) < 100:
        return "tool_specialist"

    # デフォルト: 知識モデル
    return "knowledge_specialist"
```

これは原始的なルールベースだが、実用上は大半の典型的なケースをカバーできる。RouteLLMが使うlearned routerは、このルールベースを学習済みモデルで置き換えたものだ。

### LLMベースルーター

```python
def llm_router(message: str) -> str:
    """4Bモデル自身がルーティング判断"""
    routing_prompt = f"""You are a request router. Classify into exactly one category:
- tool: needs external tool/API (weather, search, calculate)
- code: code generation, debugging, review
- knowledge: factual question, explanation, reasoning, summary

Request: {message}
Category:"""

    response = call_resident_model(routing_prompt, max_tokens=10)
    category = response.strip().lower()

    routing_map = {
        "tool": "tool_specialist",
        "code": "code_specialist",
        "knowledge": "knowledge_specialist",
    }
    return routing_map.get(category, "knowledge_specialist")
```

4Bモデルのルーティング精度はツール呼び出しベンチマーク（別記事）で実測済み: Qwen3.5-4BはLM Studio + Q4_K_M環境で97.5%の精度でタスクを正しいツールに振り分けられた。ただしこの数値はツール依存性があり、llama.cppネイティブ等では異なる可能性がある。ルーティングはfunction callingの一種なので、同等の精度が期待できるが、環境依存に注意。

## 概算: シングル vs マルチの体感差

RTX 4060 8GBでの概算値（構成Aベース、実測t/sからの計算）:

| タスク | 32B単体 (ngl=60, ~10.8 t/s) | マルチモデル | 倍率 |
|-------|------|------------|------|
| function calling | 1.4秒 (15トークン) | 0.3秒 (4B, 推定~50 t/s) | ~4.6x |
| 短い質問応答 | 5-9秒 (50-100トークン) | 1-2秒 (4B) | ~5x |
| コード生成 (200行) | 56秒 (600トークン) | 30-40秒 (14B部分オフロード)※ | ~1.5x |
| 長文要約 | 30秒+ | 25秒 (14B) | ~1.2x |

※コード生成は14Bで品質低下の可能性あり。14Bの部分オフロード速度は推定値。

タスク分布で加重平均すると: (35%×4.6 + 20%×5 + 25%×1.5 + 15%×1.2 + 5%×1) ≈ 3.2倍。概算で約3倍速くなる。ngl=60で全力運用した32Bとの比較なので、この差は控えめだが、function calling等の短タスクでの体感差は大きい。あくまで概算であり、実際のワークロードで変わる。モデル切替のロード時間（10-15秒）がオーバーヘッドになるため、頻繁に切り替えると逆効果。

モデル切替のブレークイーブン: ロード時間は10-15秒。32Bで短いタスクを処理するのは2-3秒。つまり同じモデルで5タスク以上連続処理できるならマルチが得、1タスクごとにモデルが変わるなら損。実践的にはタスクをバッチ化して同種のリクエストをまとめる。

## モデル切替を最小化するバッチ戦略

```python
from collections import deque
import time

class BatchRouter:
    def __init__(self):
        self.queues = {
            "tool": deque(),
            "code": deque(),
            "knowledge": deque(),
        }
        self.current_model = "tool"  # 常駐モデル
        self.batch_timeout = 5.0     # 秒

    def add_request(self, message: str):
        category = simple_router(message)
        self.queues[category].append(message)

    def process_batch(self):
        """最もキューが溜まっているカテゴリから処理"""
        # 現在のモデルのキューを優先 (切替コスト回避)
        if self.queues[self.current_model]:
            return self._process_queue(self.current_model)

        # 最大キューに切替
        max_category = max(self.queues, key=lambda k: len(self.queues[k]))
        if self.queues[max_category]:
            self._switch_model(max_category)
            return self._process_queue(max_category)

    def _switch_model(self, category):
        if category != self.current_model:
            # モデルアンロード → ロード (10-15秒)
            unload_model(self.current_model)
            load_model(category)
            self.current_model = category

    def _process_queue(self, category):
        results = []
        while self.queues[category]:
            msg = self.queues[category].popleft()
            results.append(generate(msg))
        return results
```

## 8GBマルチモデルの判断基準

```
マルチモデルが有効な場合:
  ✓ タスクの種類が多様 (ツール + コード + 知識)
  ✓ 短いタスクが多い (function calling, 分類, QA)
  ✓ レスポンス速度が重要
  ✓ Agentic RAGのようにツール+検索+生成を組み合わせる

シングルモデルが有効な場合:
  ✓ 長文生成が主 (小説、レポート)
  ✓ タスクの種類が単一
  ✓ モデル切替のレイテンシが許容できない
  ✓ 最高品質の知識回答が必要
```

どちらかが絶対的に優れているわけではない。自分のタスク分布を1週間ログに取ってから判断するのが正しい。60%以上が短いタスクなら、マルチモデルに移行する価値がある。
